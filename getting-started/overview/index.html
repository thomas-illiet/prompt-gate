    Overview¶

    Prompt Gate is a control plane that allows you to safely expose LLM providers (OpenAI, Anthropic, Ollama) to teams and applications.

    What is it for?¶

    Prompt Gate centralizes LLM access management:

    • Single sign-on via OIDC (Keycloak or other provider)
    • Role management: none, user, manager, admin
    • API tokens: creation, revocation, automatic expiration
    • LLM proxy: routes requests to providers with validation
    • Firewall: IP/CIDR rules, global or per service account
    • Usage monitoring: dashboards with KPIs, prompt history, estimated costs
    • MCP support: routing to MCP servers (Model Context Protocol)

    Two separate worlds¶

    Prompt Gate clearly separates two types of traffic:

    Browser traffic (Admin UI)¶

    User → Browser → API (port 8080) → OIDC → Session → PostgreSQL
    • Authentication via OIDC (code flow + PKCE)
    • Sessions stored in Redis (HTTP-only cookie)
    • Roles: none, user, manager, admin
    • Access to admin interface, configuration, dashboards

    LLM traffic (Proxy)¶

    SDK Client → Proxy (port 8081) → Token validation → Firewall → Provider
    • Authentication via Prompt Gate API tokens (JWT)
    • Validation via Redis cache
    • Firewall rules enforcement
    • Routing to OpenAI, Anthropic or Ollama
    • Usage recording (tokens, prompts, tools)

    The 5 components¶

    Component Command Description
    API promptgate api HTTP API, OIDC login, admin endpoints, frontend hosting
    Proxy promptgate proxy LLM proxy, token auth, firewall, provider/MCP routing
    Worker promptgate worker Consumes usage events from Redis, persists to PostgreSQL
    Scheduler promptgate schedule Scheduled jobs: token cleanup, access expiration, monitoring, usage cleanup
    Migrate promptgate migrate Applies GORM database migrations

    One binary, 5 commands. Each command has a narrow responsibility.

    Data stores¶

    PostgreSQL — durable data¶

    • Users and service accounts
    • API tokens (hashed) + metadata
    • Firewall rules (global and scoped)
    • Provider definitions (encrypted API keys)
    • MCP server definitions (encrypted sensitive headers)
    • Prompt history and usage (tokens, tools, interceptions)
    • Daily aggregated dashboard KPIs

    Redis — cache and real-time¶

    • Browser sessions and OIDC authorizations
    • API token auth cache
    • Provider, MCP server, and firewall snapshots
    • Version counters and hot-reload events
    • Usage event stream (promptgate:usage:events)

    Hot-reload¶

    No restart is required after a configuration change.

    flowchart TD
    AdminAPI["Change via Admin API"] --> DB["PostgreSQL"]
    DB --> Event["Redis version + Event"]
    Event --> Proxy["Proxy subscribe"]

    Proxy --> Type{"Domain"}
    Type -->|firewall| Snapshot["Refresh firewall snapshot"]
    Type -->|providers/mcp| Bridge["Rebuild AIBridge"]
    Type -->|auth| Cache["Invalidate auth cache"]
    • Firewall: snapshot refreshed instantly
    • Providers/MCP: bridge rebuild (250ms debounce)
    • Auth tokens: cache invalidation via version bump

    Security at a glance¶

    Concept Mechanism
    Browser auth OIDC + PKCE + ID token + JWKS validation
    Proxy auth JWT signed with PROMPTGATE_JWT_SECRET
    Sessions Stored in Redis, HTTP-only cookies
    Roles none (no access), user, manager, admin
    API tokens SHA-256 hash in DB, JWT for transmission
    Provider keys AES-256-GCM encrypted with PROMPTGATE_SECRETS_KEY
    Firewall IP/CIDR, first-match-wins, allow or deny
    CORS Configurable origins via PROMPTGATE_CORS_ALLOWED_ORIGINS
